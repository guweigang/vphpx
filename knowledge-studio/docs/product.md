# Product Spec

## 1. Product Definition

`Knowledge Studio` 不是一个泛化的“AI 助手 demo”，也不是一个传统的知识库后台。

它要做的是一个聚焦场景的 SaaS：

`面向专业团队的多租户知识运营与发布 SaaS`

一句话定义：

`让一个团队把内部协作维护的知识，发布成统一品牌、可订阅、可持续运营的外部知识服务。`

## 2. Primary Scenario

我们只聚焦一个主场景：

`专业服务团队把知识供给运营成对外服务。`

典型团队包括：

- 咨询 / 研究 / 顾问团队
- 金融运营 / 清算 / 风控支持团队
- 企业内部共享服务团队（HR / 财务 / 法务 / IT 支持）
- 垂直行业内容团队或专家团队

这些团队的共同问题不是“缺一个文档系统”，而是：

- 多个人在维护知识，但对外口径不统一
- 高价值知识散落在文档、FAQ、经验回答里
- 外部用户或内部服务对象会反复问相同问题
- 团队需要把知识变成标准化、可持续运营的服务入口

## 3. Who Uses It

### 3.1 Buyer

真正会决定采购或推动落地的人：

- 业务负责人
- 知识运营负责人
- 客户成功 / 交付负责人
- 中小组织 owner

他们买的不是“一个聊天框”，而是：

- 团队知识的统一服务出口
- 更低的重复答疑成本
- 更稳的品牌表达和服务一致性
- 一个可继续演进成订阅产品的知识运营底座

### 3.2 End Users

实际每天使用系统的人分两类：

- 供给侧用户：tenant owner、knowledge editor
- 需求侧用户：subscriber / visitor

## 4. Core Value Proposition

### 4.1 For Tenant Teams

- 多位知识工作者可以协作维护知识
- 知识不是停留在后台，而是被发布成对外服务
- 外部用户看到的是统一品牌，而不是个人回答风格
- 团队可以从真实提问和订阅意向中反推知识缺口

### 4.2 For Visitors / Subscribers

- 能快速看到这个知识服务解决什么问题
- 能先浏览公开页，再进入内容详情或助手入口
- 能获得结构清楚、来源明确的知识内容
- 能在需要时进一步订阅或申请访问

### 4.3 For VPHPX / VSlim

- 证明 `VSlim` 能承载真实 SaaS 结构，而不是只做 hello world sample
- 证明 `View + Session + Auth + DB + CLI + Jobs + Live UI` 可以形成完整产品闭环
- 让这个项目既能作为 flagship sample，也能作为未来 SaaS 雏形继续演进

## 5. What It Is Not

为了保持聚焦，我们明确不把它定义成下面这些东西：

- 不是通用企业知识管理平台
- 不是面向所有行业的万能 AI Portal
- 不是先做支付和计费的订阅系统
- 不是重型文档协同编辑器
- 不是大而全的 agent orchestration 平台

## 6. Core Workflow

这个产品只服务一条必须闭环的核心链路：

1. 团队观察订阅用户、访问者问题和外部反馈
2. 团队补充来源材料与标准答案
3. 团队组织版本并发布到公开面
4. 外部用户访问品牌页、阅读内容、进入知识入口验证
5. 外部用户留下订阅意向、问题、意见或新的服务信号
6. 团队再根据这些信号继续补知识、调整品牌表达和下一次发布

如果只能走到第 3 步或第 4 步，而不能回流到下一轮内容供给，这个产品就还没有真正成立。

如果一个功能不服务这条链路，就不应该优先做。

## 7. Product Surfaces

### 7.1 Console

给供给侧团队使用。

它的目标不是“后台功能很多”，而是让知识团队/产品团队稳定完成六件最重要的事：

1. 看订阅用户，关注他们的问题、意见和互动
2. 看文章与知识内容的被阅读、被提问、被验证情况
3. 看文章和标准内容的创作、整理与发布
4. 看成员协作度和贡献度
5. 设置品牌形象、团队名称和公开表达
6. 分享和传播品牌站点

所以控制台的长期结构，不应该先从技术模块出发，而应该从这六类经营动作出发。

更准确地说，控制台是一个：

`知识供给运营台`

而不是传统意义上的后台管理页。

### 7.2 Public Pages

给外部访问者使用。

它也只做两件事：

- 品牌页：解释价值、承接订阅、证明内容供给真实存在
- 知识入口：让访问者直接浏览内容，并在后续通过助手入口验证知识服务是否真的有用

## 8. ICP

当前最适合的 ICP 是：

`10-200 人规模、已经有稳定知识供给、希望把知识标准化服务化的专业团队`

不优先服务的对象：

- 纯个人创作者
- 没有稳定知识供给的小团队
- 只想要内部 wiki 的组织
- 一开始就要求复杂采购、计费、合规闭环的大企业

## 9. Why This Is More Than a Demo

它当然可以是 `VSlim` 的 flagship sample，但不能只停在 sample 心智。

更准确地说，它应该是：

`一个可公开展示、可持续迭代、具备 SaaS 方向感的产品原型。`

判断它是不是“只是 demo”，看三个标准：

### 9.1 If It Is Only a Demo

- 页面只是在展示框架能力
- 功能之间没有真正业务闭环
- 用户看完不知道谁会买、谁会用

### 9.2 If It Is a Real Product Prototype

- 有清晰目标用户
- 有清晰主流程
- 有供给侧和需求侧闭环
- 每个页面都能回答“它在这条业务链里负责什么”

`Knowledge Studio` 应该走第二条。

## 10. Current Product Boundary

当前阶段我们只做“知识供给侧 SaaS”最小可成立闭环。

### 10.1 Must Be Clear

- 多租户
- 角色边界
- 文档 / FAQ / 条目维护
- 发布版本
- 品牌页
- 验证页（增强层入口，可先轻量保留）
- 订阅入口
- 反馈回流到内容补充

### 10.2 Can Stay Lightweight

- 支付
- 真正复杂的配额结算
- 企业级 SSO
- 高级工作流审批
- 向量平台细节

## 11. Product Strategy for Next Phase

下一阶段不再横向铺功能，而是沿着主场景纵向做深。

优先级应该是：

1. 把供给侧工作流彻底讲清楚
2. 把公开页和知识入口做成可信的对外产品面
3. 把“问题反馈 -> 内容补充 -> 再发布”闭环做扎实
4. 最后再补更重的订阅、支付和高级权限

## 12. Product Test

后续每做一个功能，都先问这四个问题：

1. 它服务的是供给侧，还是需求侧？
2. 它是不是让“知识变服务”的路径更顺？
3. 它是不是强化了多租户 SaaS 的真实感？
4. 如果现在不做，会不会影响主流程成立？

如果四个问题里答不上两个以上，就先不做。
