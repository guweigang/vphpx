# Delivery Plan

## 1. Planning Principle

后续开发不再按“功能模块越多越好”推进，而是按一条必须闭环的主业务链推进：

`用户/订阅反馈 -> 内容供给 -> 发布 -> 公开页承接 -> 内容消费/轻量验证 -> 新问题与新反馈 -> 再补知识`

任何任务如果不能加强这条链路，就降级。

这条闭环再拆开，就是供给侧团队每天真正要经营的六件事：

1. 看订阅用户、问题和意见
2. 看内容是否被阅读、被提问、被验证
3. 做创作、编辑和发布
4. 看团队协作和贡献
5. 维护品牌表达
6. 做公开传播和站点分享

如果某个页面或功能不能回到这六件事之一，就说明它偏离了主场景。

## 2. Current Stage

当前项目已经不是从 0 开始的 skeleton，而是处在：

`从框架示例走向产品原型的收敛阶段`

当前已经有的基础：

- 多租户 workspace
- 登录与角色
- 内容库（文档 / 条目）
- 发布中心
- 公开品牌页
- 公开知识入口
- 中英双语
- 真实数据库写路径

当前最大问题不是“缺功能”，而是：

- 产品叙事还不够收敛
- 页面职责有时还不够清晰
- 供给侧与需求侧闭环还需要更顺

## 3. Next Milestones

### Milestone A: Product Narrative Lock

目标：把所有页面都收敛到一个明确的 SaaS 场景。

交付：

- 收敛产品定义
- 收敛信息架构
- 明确每个页面在主流程中的职责
- 去掉与主场景无关的表达

完成标准：

- 我们能用一句话讲清楚产品做什么
- 每个主要页面都能回答“它在闭环里负责什么”

### Milestone B: Supply-Side Workflow Hardening

目标：把供给侧工作流做成真正可演示、可回流的主线。

交付：

- 文档 / FAQ / 条目编辑体验继续打磨
- 发布中心继续强化“版本载荷 + 发布理由 + 预览”
- 反馈问题可直接回流成内容草稿

完成标准：

- 一个租户团队可以完整演示“看反馈 -> 补内容 -> 发布 -> 对外查看 -> 再回流”

### Milestone C: Public Conversion and Validation

目标：把公开页和知识入口做成可信的外部产品面。

交付：

- 品牌页承接订阅意向
- 公共详情页串起内容证明链
- 验证页保留为轻量验证入口，不作为当前主叙事

完成标准：

- 外部用户能看懂“这是什么服务”
- 外部用户能自然进入浏览、订阅或轻量验证动作

### Milestone D: SaaS Readiness Layer

目标：给后续 SaaS 化留出清晰接口，但不提前做重。

交付：

- 套餐模型继续规范
- 成员 / 品牌 / 公开面配置继续结构化
- 测试边界补齐

完成标准：

- 代码库可继续作为开源项目分享
- 也可继续往真实 SaaS 演进

## 4. Immediate Backlog

接下来建议只做下面几类任务：

1. 调整页面职责与信息架构
2. 打磨内容维护和发布工作流
3. 强化公开页与内容详情页的承接关系
4. 补回归测试，稳住当前基础

明确暂缓：

- 支付
- 复杂计费
- 高级 agent
- 过重的工具平台能力
- 泛化行业场景扩张

## 5. Definition of Done for This Phase

这一阶段完成，不看“功能数量”，只看下面几点：

- 能清楚讲明白产品定位
- 能完整演示一个团队如何把知识运营成外部服务
- 能展示外部访问者如何进入公开页、提问、订阅、留下反馈
- 能展示问题反馈如何真实反推内容补充和下一次发布
- 页面结构和叙事不再发散
