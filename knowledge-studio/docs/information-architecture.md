# Information Architecture

## 1. One-Line Model

`Knowledge Studio` 的信息架构不应该围绕“后台有什么功能”组织，而应该围绕这条业务链组织：

`团队生产知识 -> 团队发布知识 -> 外部用户看到品牌页 -> 外部用户进入助手提问 -> 团队根据反馈继续补知识`

所以页面职责必须服务这条链路，而不是各自孤立存在。

## 2. Two-Sided Product

产品天然分成两侧：

### 2.1 Supply Side

给 tenant owner / knowledge editor 使用。

他们关心的是：

- 现在要补什么内容
- 哪些内容已经可发布
- 这次发布会影响什么外部体验
- 外部用户最近在问什么

### 2.2 Demand Side

给 visitor / subscriber 使用。

他们关心的是：

- 这是什么服务
- 我能得到什么价值
- 能不能马上试一下
- 如果有价值，怎么继续订阅或申请访问

## 3. Primary Navigation

如果按产品团队真正关心的事情来组织，控制台主导航不应该只是“工作台 / 内容库 / 设置”这类后台词汇。

更合理的一级模型应该是六类经营动作：

- 用户与订阅
- 内容表现
- 创作与发布
- 团队协作
- 品牌设置
- 传播分享

这不是按技术模块分，而是按团队每天在经营什么分。

当前阶段不一定马上把导航文案全部改成这六项，但页面职责和后续重构都应该朝这六类动作收敛。

## 3.1 Six Core Jobs

### 用户与订阅

团队最先应该看到的是：

- 最近新增了哪些订阅线索
- 订阅用户最近问了什么
- 哪些问题值得跟进
- 哪些用户反馈应该回流成内容补充

### 内容表现

团队需要看到的不是泛数据，而是内容是否真的被使用：

- 哪些文档被读了
- 哪些标准答案被命中了
- 哪些问题反复出现
- 哪些回答虽然被展示，但仍然不够有说服力

### 创作与发布

这是当前产品最核心的供给链路：

- 补来源材料
- 补标准答案
- 组织本次发布载荷
- 验证公开页和助手页是否承接成功

### 团队协作

团队不是单人写作工具，所以要能回答：

- 谁最近在补内容
- 谁在维护 FAQ / 条目
- 谁负责发布
- 当前协作是否活跃、是否失衡

### 品牌设置

这个系统不是只面向内部，所以品牌表达必须是一级能力：

- 团队名称
- 品牌名
- 公开页文案基调
- 助手命名与对外身份

### 传播分享

品牌页不是摆设，而是传播入口：

- 如何分享品牌页
- 如何分享助手页
- 如何拿订阅方案页去承接外部访问
- 如何把公开内容作为销售、交付、客户成功的辅助材料

## 4. Console Responsibilities

### 4.1 `/console`

角色：

`供给侧总览页`

它负责回答：

- 今天团队先做什么
- 哪些内容在待补
- 最近有哪些外部信号值得处理
- 当前发布状态是否健康
- 最近哪些订阅用户和访问者值得跟进
- 哪些内容真的被使用、被提问、被验证

它不应该变成：

- 冗长的数据看板
- 另一个内容列表页
- 另一个发布页

### 4.2 `/console/knowledge/documents`

角色：

`长文档供给页`

它负责：

- 维护可作为知识来源的正式文档
- 承接“需要补原始材料”的内容需求
- 支持编辑、预览、发布前整理

它不是：

- 所有内容的总入口
- 对外展示页

### 4.3 `/console/knowledge/faqs`

角色：

`标准答案供给页`

它负责：

- 维护 FAQ / 条目 / 标准回答
- 承接来自助手缺口的快速补答
- 形成更容易被问答系统命中的短内容

它和文档页的区别必须长期清楚：

- 文档偏来源材料
- 条目偏标准回答

### 4.4 `/console/releases`

角色：

`发布与上线关口`

它负责：

- 选择本次公开版本载荷
- 记录本次发布理由
- 预览本次发布会进入哪些公开页内容
- 把“知识补充”正式转成“对外服务更新”

它不是：

- 普通列表页
- 审计页
- 内容编辑页

### 4.5 `/console/members`

角色：

`供给侧协作控制面`

它负责：

- 谁能参与知识供给
- 谁能发布
- 团队协作边界是什么

它的价值是支撑“团队维护”，不是孤立的组织管理功能。

### 4.6 `/console/ops`

角色：

`最低必要的运维与审计页`

它负责：

- 看任务
- 看审计
- 支撑 demo 与产品原型的可信度

它不应该膨胀成完整平台运维中心。

## 5. Public Responsibilities

### 5.1 `/brand/{tenant}`

角色：

`品牌承接页`

它负责：

- 先让访问者理解“这是什么知识服务”
- 证明当前供给是真实存在的
- 把访问者引向方案选择、订阅申请、助手体验

它不是：

- 内容库镜像
- 冗长的营销官网

### 5.2 `/brand/{tenant}/assistant`

角色：

`价值验证页`

它负责：

- 让访问者马上提一个真实问题
- 展示答案、引用和可信度
- 暴露当前知识缺口，形成供给侧回流线索

它不是：

- 泛化的“产品介绍页”
- 只有解释没有行动的页面

### 5.3 `/brand/{tenant}/documents/{id}` and `/entries/{id}`

角色：

`内容证明页`

它负责：

- 证明这个助手背后有真实内容
- 让引用链条可追踪
- 强化公开页与助手页的可信度

它不是：

- 独立 CMS
- 复杂内容中心

## 6. User Flow by Role

### 6.1 Tenant Owner

典型路径：

1. 打开工作台
2. 看本周要补的内容和发布状态
3. 到发布中心组织本次公开版本
4. 打开公开页验证结果

### 6.2 Knowledge Editor

典型路径：

1. 从工作台或助手反馈进入内容库
2. 补 FAQ 或补文档
3. 编辑内容并形成可发布材料
4. 交给 owner 或在允许范围内继续发布

### 6.3 Visitor / Subscriber

典型路径：

1. 进入品牌页
2. 理解服务定位与方案
3. 查看公开内容证明
4. 进入助手页提问
5. 提交订阅意向或继续访问

## 7. What Should Be Removed from Our Thinking

为了避免继续发散，后续不要再把它想成：

- 一个功能很多的后台
- 一个“顺便加点 AI”的 CMS
- 一个先展示框架能力再找业务壳的 sample

我们应该始终把它想成：

`一个知识供给侧 SaaS，外加一个知识服务需求侧入口`

## 8. Evaluation Rule for Every Page

后面每个页面都先回答这三个问题：

1. 它服务供给侧还是需求侧？
2. 它在主链路里的前一步和后一步分别是什么？
3. 如果删掉它，主链路会不会断？

如果答不清，这个页面就还没设计好。

## 9. Current Optimization Priorities

按信息架构看，当前最应该继续优化的是：

1. 工作台
让“今天做什么、用户最近在问什么、哪些内容最值得跟进”更强，不再像混合仪表盘。

2. 内容库
让文档和 FAQ 的边界更清楚。

3. 发布中心
让“为什么发布、带什么出去、发布后去哪看”更清楚。

4. 品牌页
让订阅承接、品牌表达和传播入口更顺。

5. 助手页
让提问、答案、引用成为绝对主角。

6. 团队与品牌经营面
把协作贡献、品牌设置、分享传播从“附属页”提升成真正的经营能力。
